这次介绍一个新玩意:httpx这个库说是既能同步又能异步,并且同步的时候还比requests快,所以这次玩玩这个库

为了不剧透,这次我就只是按我最初的思路一步一步来,中途遇到的情况中途再说

先说httpx,在同步的时候,用法和requests一样,异步的时候,只是和aiohttp有一点点不一样:虽然建立session的类(aiohttps是ClientSession,httpx是AsyncClient)都有上下文管理器,但是aiohttp里建立session之后它的实例的get,post之类的函数都是有异步上下文管理器的;而httpx的session建立起来之后它的实例的那些函数都可以直接await.我不知道他们两个的性能差异是不是就是这里的问题但是用httpx异步请求的时候要写的东西确实少多了

这个 json 的结构好像是这样的(括号内为数据类型,这里只列我们现在可能需要的,并且以下都是我的猜测):

  • code: 是否成功,0 则是(int)
  • data: 所有评论(dict)
    • cursor: 标识(?)(dict)
      • all_count: 总评论数(int)
      • is_end: 是否为最后一页(bool) *
    • replies: 评论(list)
      • like: 点赞数(int)
      • rpid: 该评论对应编号(int)
      • member: 成员信息(dict)
        • mid: 用户 id(str)
        • uname: 用户名(str)
        • sex: 性别(可能用不到)(str)
        • sign: 签名(可能用不到)(str)
        • avatar: 头像链接(可能用不到)(str)
        • level_info: 等级信息(dict)
          • current_level: 当前等级(int)
      • content: 评论内容(dict)
        • message: 评论内容(str)
      • replies: 评论回复,其下结构参考上文(list)

然后就遇到了一个新的问题:每次获得的评论只有20条并且每个根评论的子评论最多3条.又是漫长的查资料的过程

怎么只有20条根评论?

功夫不负有心人,我看到了这篇文章:python爬B站评论（一通百通） - KledKled - 博客园 (cnblogs.com),虽然里面的api现在用不成了,但是他的思路启发了我,我需要再好好分析一下需要那些api了

最开始的时候我们说到评论的api需要的参数硬是被我砍到了只剩两个,但是这样我们拿到的根评论个数只有20个,所以我就从第21个根评论开始找,然后发现了这个东西:

https://api.bilibili.com/x/v2/reply/main?csrf=87d9440b8e6b504fdadd12ea0619f789&mode=3&next=0&oid=59359056&plat=1&seek_rpid=&type=1

这是直接复制粘贴过来的地址,我们要找的就是那个被我们忽视了的参数.可以发现里面有一个next参数,在试验时候发现这确实是指定页数的参数,所以我们可以把每一页的内容都拿到手了

一个新的问题出现了:一共多少页?

比如截至2022-12-20,视频BV1Za41157Sp共有1884条评论,那么页数就是[1884/20]+1

但是这个想法有个问题:有种东西叫子评论,一页20个评论是根评论的数量,这样页数就太多了,遂放弃

让它不要这么散

之前说如果直接用notebook运行整个程序那就实在有太多冗余了,不说其他思路,就连打开这个notebook都比较困难(16.2MB,内存受不了),所以在整理了一些东西之后,把它封装成了一个类,这样就可以一次只用输入一个链接就可以直接包揽所有如bvid,cid等内容的获取而不需要分函数导致每个函数都要带上一个长长的字符串或者一个变量之类的看上去并不怎么优雅的操作

既然「万物皆对象」那索性就把「对象」的思想再深刻落实一下

这玩意我就直接放到github上了Bilibili-Native-API-Encapsulation

Post Script

看看整个过程,把这些东西封装成这样,搞得让刚来的人感觉这么复杂,阿b也是费心了

回顾下来整个过程,我突然萌生一个想法:也许可以用异步同时请求评论和弹幕来节约时间,不过这样可能任务调度起来会有些麻烦,有空再考虑这样优化一下吧

或者,如果只是想看一眼有那些评论,仅打印到屏幕上而不多占用内存的话,既然b站api默认是用的页的形式来获取评论,那我们也可以用这一点来节约内存(就像流一样),不过这样想,「流」的这一想法好像可以不止用于拿来看,也可以用于写入文件,这样确实更节约内存(原先的模型为了能分析内容所以直接一次性全获取了).并且写入文件的时候也可以用类似于pandas的库
